Webpack 缓存策略

Long Term Caching

为了有效缓存文件，在URL中应该有一个hash或版本。我们就可以在output文件夹中emit或move一个版本为V1.3的文件夹。但这是几个缺点：开发的额外的文件和未更改的文件不能从缓存中获取。

Webpack可以为文件的文件名添加hash值。Loaders已经做到了这些。对于chunks，我们必须让它可用。这里有两种级别：

  • Compute a hash of all chunks and add it：计算所有chunks的hash值然后添加
  • Compute a hash per chunk and add it

One hash for the bundle

通过在配置选项中给文件名添加[hash]来激活。

One hash per chunk

通过在配置文件中给chunk filename添加[chunkhash]来激活。

需要在HTML文件中引入这个具有hash的entry chunk。

为了保证Hot Code Replacement的功能，建议使用One hash for the bundle。

Get filenames from stats

我们可能想获得asset中最终的文件并把它添加到HTML中。这个在webpack stats是可以实现的。如果使用CLI，可以添加–json并运行它就能获取到stats的JSON格式。

我们可以在配置中添加一个插件，如assets-webpack-plugin，该插件允许读取stats对象。

stats JSON中有一个关键属性assetsByChunkName，它是一个对象，chunk name是key，filename(s)是value。

Webpack code splitting

Webpack Code Splitting

对于大型应用来说，将所有的代码都打包到一个文件中一次加载不是很明智的做法，尤其是在某些代码块只需要在特定情况下加载的时候。Webpack可以将你的代码按需分割成“chunks”加载。其他的一些“bundlers”被成为“layers”、“rollups”或“fragments”。这个特性被成为代码分割（code splitting）。

这是一个可选的功能。你可以在代码中定义自己的分割点。Webpack的任务只关心依赖关（dependencies）系、输出文件（output files）和运行时的东西（runtime stuff）。

Code Splitting不只是提取共同的代码放到一个公共的文件中，更重要的功能是拆分代码，按需加载。这可以保证应用程序首次请求时下载的是最小的和所需求的文件。

Defining a split point

AMD和CommonJs都可以通过不同的方式按需加载代码。两个都支持分割点。

require.ensure (CommonJs)

require.ensure方法保证了当调用callback的时候，dependencies中的每一个dependency都可以异步加载。callback调用时需要require方法作为参数。

require.ensure只加载模块，并不执行它们。

require (AMD)

AMD规范定义了一个require的异步方法。

当require被调用的时候，所有的dependencies都被加载，callback被调用时依赖所有被加载的dependencies。

AMD require方法加载并执行这些modules，在webpack modules中从左向右执行；require方法可以不加回调函数。

Chunk content（chunk内容）

所有的dependencies都是在split point处生成新的chunk。Dependencies也都是递归添加。

如果将函数表达式（或绑定函数表达式）作为回调函数传递给split point，Webpack也会自动在这个函数表达式中通过所需的依赖（dependencies require）生成chunk文件。

Chunk optimization（chunk优化）

  • 如果两个chunks包含相同的modules， 它们会被合并成一个。这可能会导致chunks有多个parents
  • 如果一个module在一个chunk的所有parents中都是可用的，它应该从这个chunk中删除
  • 如果一个chunk包含了另一个chunk的所有modules，这个chunk应该被保留，因为它能满足多个chunks

Chunk loading（chunk加载）

根据目标（target）运行时逻辑的配置选项，加载chunk并添加到bundle中。对于web target chunks是通过JSONP加载。一个chunk只能加载一次，并行请求被合并成一个。运行时检测所加载的chunks是否满足多个chunks。

Chunk types

Entry chunk

一个entry chunk（入口块）包含运行时的多个modules。如果chunk包含模块0就在运行时执行它，如果没有，则会等待chunks中包含了模块0后在执行它（每次当chunk中包含module 0）。

Noraml chunk

normal chunk不包含runtime，它仅包含一组modules，结构取决于chunk的加载算法。JSONP模块被封装在jsonp的回调函数中。chunk还包含chunk id的一个列表。

Initial chunk(non-entry)（初始chunk，非入口）

一个initial chunk是一个normal chunk。唯一的区别是：优化（optimization）将其来说更重要，因为它要计算初次加载的时间（像entry chunks）。chunk type可结合使用CommonsChunkPlugin。

Split app and vendor code

把应用分成两个文件，如index.js和vendor.js，我们可以require这个vendor文件在vendor.js中。然后将这个名字传递给CommonsChunkPlugin。

这将会从应用程序的chunk中删除掉所有在vendor chunk中的modules。bundle.js文件将不会包含任何依赖关系，只包含应用程序代码。其他的都在vendor.bundle.js中。

在HTMl中，要先引入vendor.bundle.js，然后在引入bundle.js。

Multiple entry chunks（多入口chunks）

可以配置多个entry points，因此就会有多个entry chunks。运行时会包含这个entry chunk，但在运行时一个页面只能包含一个entry chunk（也有例外情况）。

Running multiple entry points（运行多个entry points）

随着运行时CommonsChunkPlugin被移到commons chunk，这个entry point现在是initial chunks。然而只有一个initial chunk，多个entry chunks已经被加载，这样暴露了一个页面上可以运行多个entry points的可能性。

Commons chunk

CommonsChunkPlugin插件可以把出现在多个entry chunks中的modules移到一个新的entry chunk（commons chunk）中，运行时也可以移到commons chunk中。它的意思就是说原来旧的entry chunks就是现在的initial chunks。请参考所有的list of plugin。

Optimization

一些优化的插件根据标准块进行合并。

  • LimitChunkCountPlugin
  • MinChunkSizePlugin
  • AggressiveMergingPlugin

Named chunks

require.ensure函数接收第三个参数，必须是个字符串。如果两个split point传递相同的字符串，那它们就使用相同的chunk。

require.include

require.include是Webpack的专用函数，用来在当前chunk中添加新的module，但不执行该module（该语句在bundle中会被移除）。

如果一个module有多个child chunks的话，require.include函数是比较有用的。require.include父级会包含这个module，chunk中的modules实例会在chunks中消失。

async函数就是将Generator函数的星号换成了async，将yield换成了await而已。

async函数对Generator函数有以下三点的改进。

  • 内置执行器。Generator函数的执行必须依靠执行器，所以才有了co模块，而async函数自带执行器。
  • 更好的语义。async和await比星号和yield语义清楚很多。
  • 更广的适用性。co模块约定，yield命令后面只能是Thunk函数或Promise对象，而async函数的await后面，可以是Promise对象和原始类型

需要注意的几点

  • await后面的promise对象，运行结果可能是reject，所以将await放在try catch代码块中
  • await只能用在async函数中，如果在普通函数中，就会报错
  • 如果希望多个请求并发执行，使用Promise.all方法

JavaScript之Ajax跨域

Ajax跨域请求

Jsonp

Jsonp可以解决XmlHttpRequest请求不能跨域的限制，原理是通过动态创建一个script元素来请求一段js脚本，让这段脚本在页面作用域里执行，迂回实现类似Ajax的请求。

Jsonp的缺陷：只能是GET请求，不能使用POST请求，请求内容长度有限制，没有XHR的状态，服务器端无法获取用户信息。

CORS(Cross-Origin Resource Sharing)

CORS是W3C解决跨域请求的一个规范。被请求的服务器需要设置Access-Control-Allow-Origin: 允许请求的域。

请求过程：发送方XHR请求报头里有一条Origin:请求页面的域，被请求服务器设置根据Access-Control-Allow-Origin设置判断是否允许该请求，如果允许会把内容返回给请求页面。

详情可参考MDN文档：HTTP access control CORS。

服务器代理请求

页面发送请求给本域服务器的一个代理脚本，代理脚本完成对其他域的请求，然后把结果返回给客户端。优点是可以模拟GET或POST请求方法，并把参数和客户端Cookies传递过去，缺点是实施起来比较麻烦。

不同域之间的JavaScript交互

document.domain

只能解决一个域名下的不同二级域名页面跨域，如bzn.zbj.com和yxl.bzj.com，设置document.domain=’zbj.com’。

window.name

window.name一般用来获取子窗口：window.frames[windowName]。一个窗口无论加载什么页面，window.name都保持不变。

window.postMessage

window.postMessage出现在HTML5规范中，可以实现脚本跨域通信。当window.postMessage方法被调用时，目标窗口的message事件被触发，就可以实现不同域之间的信息交流。

Iframe

页面Iframe呈树形嵌套，它的特点如下：

  • 任何一个节点窗口都能获取树上其他窗口的引用。子iframe可以通过top／parent来引用父窗口，父窗口可以通过window.frames[0]或window.frames[name]来引用子窗口
  • 树上不同源的窗口不能获取对方的内部状态
  • 树上任何窗口可以设置(不能读取)其他窗口的location/url
  • 一个页面URL的锚点(http://www.bzn.com/index.html#bai中的#bai)改变不会引起页面刷新

根据以上特点，Iframe进行跨域交互的方法：

  • 改变iframe嵌套树上的目标窗口的URL，目标窗口通过轮询来监听URL的变化，以此通过锚点链在不同iframe直接传递数据
  • 不同域的iframe不能获取内部状态，可以通过创建一个根目标iframe相同域的隐藏iframe来传递数据

Javascript之Cookie

什么是cookie

cookie是存储于访问者的计算机中的变量。每当同一台计算机通过浏览器请求某个页面时，就会发送这个cookie。

cookie是访问过的网站创建的文件，用于存储浏览信息。

在javascript中可以通过document.cookie来读取或设置这些信息。

cookie基础知识

  • cookie有大小限制，每个cookie所存放的数据不能超过4KB，如果超过，则返回空字符串
  • cookie是以文本形式存放在客户端的计算机中
  • cookie的格式：＝
  • cookie的有效期，默认的生命周期是浏览器关闭时候，如果希望在浏览器关闭后还能使用，则需要设置cookie的有效期
  • typeof document.cookie的结果是string
  • cookie有域和路径的概念，默认不能跨域访问cookie
  • 多个cookie可以存放在一个cookie文件中
